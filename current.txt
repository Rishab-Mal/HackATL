WHAT THE APP IS
---------------
A B2B textile-scrap exchange, branded "Reweave": factories photograph piles of
fabric scraps, computer vision groups them by color/material, those groups
become marketplace "lots" with AI-set pricing, and recyclers/makers claim them.
An admin dashboard shows P&L and environmental impact; buyers have a dedicated
marketplace + orders history; the AI Destination Engine recommends the best
reuse pathway per scrap group with buyer matches and environmental equivalents.

Three role portals (factory / admin / buyer) plus a public landing page for
logged-out visitors.

Stack: FastAPI backend (backend/), React + Vite frontend (frontend/), Tailwind
CSS + @tremor/react, single dev server each, /api/* proxied to backend:8000.
DB: Postgres via Supabase (DATABASE_URL in backend/.env), SQLite fallback for
offline dev. Auth: JWT in localStorage under key `reweave_token`.

Demo credentials: factory@demo.com/factory123, admin@demo.com/admin123,
buyer@demo.com/buyer123.


DATABASE / MODELS
-----------------
models.py now has 4 tables (was 3 last update):

  Lot — fabric scrap lot:
    id, name, description, fabric_type, composition, color_name, color_hex,
    lot_key (String, nullable, indexed — stable "type::comp::color" slug),
    scan_run_id (FK → scan_runs.id, SET NULL on delete),
    piece_images (JSON, list of {src, ...} dicts),
    piece_count, weight_kg, price_usd (base), carbon_saved_kg, water_saved_l,
    origin_lat / origin_lng (Float, nullable — GPS coords of the factory worker
    who scanned this lot, stamped at scan time),
    status ("available"|"claimed"|"unlisted"), claimed_by, claimed_at,
    created_at.

  ScanRun (NEW) — one camera scan that produced one or more lots:
    id, annotated_image_data_url (Text), image_width, image_height,
    piece_count, group_count, total_weight_kg, total_carbon_saved_kg,
    total_water_saved_l, summary (JSON), created_at.

  User — portal login:
    id, email (unique), password_hash (sha256), role, name,
    lat / lng (Float, nullable — buyer's saved GPS position for the
    marketplace map and shipping estimates).

  Buyer — recycler/maker marketplace profile (separate from User):
    id, name, type ("recycler"|"maker"), location, description,
    interested_materials (comma-separated).

schemas.py additions:
  - LotCreate / LotOut: now include lot_key, scan_run_id, piece_images
    (list[dict]), origin_lat, origin_lng.
  - ScanRunCreate / ScanRunOut: new (for POST /api/lots/scan-runs).
  - LotClaim: updated to include optional quantity_kg field.


BACKEND — WHAT EXISTS
---------------------

routers/lots.py — significantly refactored:
  - POST /api/lots: now does smart lot MERGING. Before creating a new lot,
    checks for an available lot with the same lot_key (or matching
    fabric_type + composition + color_name if lot_key is null) that has no
    scan_run_id. If found, merges the incoming submission into that lot:
    adds weight + piece_count, appends piece_images (capped at 24), merges
    description. This means repeated scans of the same material accumulate
    into a single listed lot rather than fragmenting into many tiny ones.
  - POST /api/lots/scan-runs: creates a ScanRun record — used by Capture.jsx
    to stamp the annotated image + summary stats for a scan session.
  - GET /api/lots: new filter parameter `claimed_by` (used by BuyerOrders to
    fetch a buyer's own claimed lots by user.name).
  - POST /api/lots/{id}/claim (CHANGED): now handles REAL partial quantity
    claims. If quantity_kg < lot.weight_kg, creates a new Lot row for the
    claimed portion (ratio-splits weight, price, carbon, water, piece_images
    proportionally), deducts from the original lot (which stays available
    with the remainder). If quantity_kg >= lot.weight_kg or null, claims
    the whole lot as before. This resolves the long-standing gap where qty
    was tracked in the cart but silently ignored at the backend.
  - DELETE /api/lots/{id}: new endpoint to permanently delete a lot.
  - PATCH /api/lots/{id}/delist and /relist — unchanged.

routers/auth.py — extended:
  - GET /api/auth/me: now returns lat/lng on UserOut.
  - NEW: PUT /api/auth/me/location — body {lat, lng}, saves the buyer's
    GPS position on their User row. Called by SupplierMap when the browser
    shares location.

routers/admin.py — unchanged functionality:
  - GET /api/admin/metrics — STILL NO AUTH (longstanding gap).
  - POST /api/admin/reset-demo — wipes lots table.

routers/destinations.py + destinations.py — unchanged (AI Destination Engine):

routers/vision.py, routers/marketplace.py, routers/chat.py — unchanged.
pricing.py, constants.py — unchanged.
seed.py — unchanged (idempotent, seeds 3 demo logins + 8 buyer profiles only;

api.js additions (new frontend functions):
